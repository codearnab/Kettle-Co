# Project Rules

## General
- Always understand the existing codebase before making changes.
- Never overwrite existing functionality unless explicitly requested.
- Ask for clarification if requirements are ambiguous.
- Work incrementally instead of generating the whole project at once.

## Planning
- Before writing code, explain the implementation plan.
- Break work into small milestones.
- Wait for approval before starting implementation.

## Code Quality
- Write clean, readable, and maintainable code.
- Follow DRY and SOLID principles where appropriate.
- Avoid duplicate logic.
- Use meaningful variable and function names.

## Folder Structure
- Keep folders organized.
- Separate components, hooks, utils, services, and assets.
- Do not create unnecessary files.

## Components
- Create reusable components.
- Avoid large monolithic components.
- Keep components focused on a single responsibility.

## Styling
- Use consistent spacing and naming.
- Make every UI responsive.
- Avoid inline styles unless necessary.

## Performance
- Minimize unnecessary re-renders.
- Lazy load when appropriate.
- Optimize images and assets.

## Error Handling
- Handle loading states.
- Handle empty states.
- Handle API errors gracefully.

## Git
- Make logical changes.
- Avoid modifying unrelated files.

## Before Every Task
Always:

1. Explain the plan.
2. Mention which files will be modified.
3. Explain why.
4. Wait for approval before writing code.

Never skip the planning phase.
